# Blue Harbor Audit Summary

## Findings

| ID | Severity | Issue | Evidence | Impact |
| --- | --- | --- | --- | --- |
| 1 | critical | The dashboard data layer is still wired to the legacy unprefixed pipeline schema instead of the `bh_` schema. | `sql/create-agency-dashboard-tables.sql:2-39`, `sql/bh-pipeline-tables.sql:28-49`, `sql/bh-pipeline-tables.sql:71-73`, `lib/dashboard.ts:6-53` | If the database is provisioned from `bh-pipeline-tables.sql`, the dashboard will query the wrong tables (`leads`, `team_members`, `activity_log`) and silently miss the `bh_clients`, `bh_team_members`, and `bh_activity_log` data model. |
| 2 | critical | The public intake API still targets legacy tables and legacy columns that do not exist in `bh_clients` / `bh_intake_submissions`. | `app/api/intake/route.ts:12-16`, `app/api/intake/route.ts:22-28`, `app/api/intake/route.ts:43-47`, `app/api/intake/route.ts:53-59`, `app/api/intake/route.ts:61-87`, `app/api/intake/route.ts:112-116`, `sql/bh-pipeline-tables.sql:47-64` | `GET /api/intake` and `POST /api/intake` will break against the `bh_` schema because they expect `clients`, `intake_submissions`, `organization_id`, `current_step`, `updated_at`, `physical_address`, `john_cell`, `craig_cell`, `direct_lending_min`, `brokered_loan_min`, and other fields that are not in the `bh_` tables. |
| 3 | critical | Proposal flows assume `bh_proposals.client_id` is a `leads.id`, but the schema defines it as `bh_clients.id`. | `sql/bh-pipeline-tables.sql:71-73`, `app/api/proposals/route.ts:26-39`, `app/api/proposals/route.ts:98-103`, `app/api/proposals/accept/route.ts:28-33`, `app/proposal/[publicSlug]/page.tsx:27-33`, `app/dashboard/clients/[clientId]/proposal/page.tsx:22-27` | Proposal creation, proposal rendering, proposal acceptance, and proposal status sync are all coupled to `leads`. In a real `bh_` deployment, those lookups and updates will hit the wrong records or no records at all. |
| 4 | critical | The proposal builder does not load intake data and does not use the schema's `bh_clients.report_id` link; it still pulls reports by `lead_id = clientId`. | `sql/bh-pipeline-tables.sql:28-31`, `app/dashboard/clients/[clientId]/proposal/page.tsx:40-48`, `app/dashboard/clients/[clientId]/proposal/page.tsx:95-101`, `components/proposal/ProposalBuilder.tsx:25`, `components/proposal/ProposalBuilder.tsx:38` | The builder cannot satisfy the intended "intake + report" assembly flow. It never queries `bh_intake_submissions`, always passes `intakeData={null}`, and ignores the explicit `report_id` field on `bh_clients`. |
| 5 | critical | The dashboard auth model is not centralized, and several dashboard pages only check "is authenticated" before reading service-role data. | `app/dashboard/layout.tsx:3-8`, `app/dashboard/stats/page.tsx:12-25`, `app/dashboard/leads/[id]/page.tsx:64-90`, `app/dashboard/leads/new/page.tsx:8-13`, `app/dashboard/clients/[clientId]/proposal/page.tsx:16-20` | Any authenticated Supabase user, even a non-team account, can reach those pages and read or operate on pipeline data because the pages use `createAdminClient()` after only `auth.getUser()`. |
| 6 | critical | `POST /api/generate-report` has no auth guard at all. | `app/api/generate-report/route.ts:15-33` | An unauthenticated caller can generate reports for arbitrary `leadId` values, trigger external scraping/AI work, mutate database rows, and send outbound email. |
| 7 | critical | Several service-role API routes only require "logged in" and do not verify team membership or admin role before reading or mutating sensitive data. | `app/api/proposals/route.ts:8-12`, `app/api/proposals/route.ts:64-68`, `app/api/lead-status/route.ts:19-46`, `app/api/leads/delete/route.ts:7-30`, `app/api/unlock-report/route.ts:8-55`, `app/api/generate-deepdive/route.ts:14-45`, `app/api/unlock-deepdive/route.ts:9-65`, `app/api/find-competitors/route.ts:10-30` | Any authenticated user can create/update/send proposals, change lead status, delete leads, unlock reports, generate deep dives, or rewrite competitor data because the handlers immediately switch to the service-role client after a bare `auth.getUser()` check. |
| 8 | high | The public intake HTTP flow bypasses the intended `bh_` RLS model and exposes service-role reads/writes keyed only by a raw `clientId`. | `app/api/intake/route.ts:4-10`, `app/api/intake/route.ts:33-41`, `app/intake/[clientId]/page.tsx:13-16`, `components/intake/IntakeForm.tsx:41`, `components/intake/IntakeForm.tsx:101-110` | Anyone with a valid intake URL can read the associated client record and upsert intake data through a service-role route. Even after the table mismatch is fixed, this flow still depends on obscurity of the UUID instead of database- or app-level authorization. |
| 9 | critical | The anon update policy on `bh_proposals` is overly broad: anon can update any proposal row with a non-null slug, not just accept it. | `sql/add-proposal-fields.sql:10-14` | PostgREST update policies are not column-scoped. With this policy, anon callers can update any writable column on any proposal that has a `public_slug`, including pricing, roadmap, title, and status. |
| 10 | high | The anon read policy on `bh_proposals` is also overly broad: it exposes every proposal row whose `public_slug` is non-null. | `sql/bh-pipeline-tables.sql:197-199` | Instead of limiting access to a single slugged proposal through app logic, the database policy allows anon enumeration of all public proposals via the data API. |
| 11 | medium | The anon insert policy on `bh_intake_submissions` accepts any row without constraining which client it belongs to. | `sql/bh-pipeline-tables.sql:169-171` | Any anon caller with an existing `bh_clients.id` can insert submissions for that client. There is no token, ownership, or route-bound check at the database layer. |
| 12 | medium | The public proposal page and acceptance endpoint bypass the intended anon RLS path by using the service-role client directly. | `app/proposal/[publicSlug]/page.tsx:15-21`, `app/api/proposals/accept/route.ts:11-22`, `sql/bh-pipeline-tables.sql:126-129` | The repo defines anon proposal policies, but the app does not exercise them. Security for the public proposal flow therefore depends entirely on the server code path and slug handling, not on the database policy model. |
| 13 | low | Email delivery is misconfigured locally because `RESEND_API_KEY` is blank, while email code assumes it exists. | `.env.local:5`, `lib/resend.ts:6-8`, `lib/resend.ts:51-85`, `lib/resend.ts:87-119`, `lib/resend.ts:121-147`, `lib/resend.ts:149-183` | Report-ready, unlock, deep-dive, and admin-notification emails will fail at runtime until a real Resend API key is configured. |
| 14 | low | There are confirmed dead/unused symbols in the current codebase. | `components/proposal/ProposalBuilder.tsx:10`, `components/proposal/ProposalBuilder.tsx:38`, `components/report/MultiComparisonTable.tsx:12`, `app/api/submit-form/route.ts:85` | These do not break runtime behavior, but they are a reliable sign that the proposal builder and report components still contain unfinished migration paths and stale code. |

## Verification Notes

- `npx eslint app lib components types --ext .ts,.tsx` completed and surfaced the dead-symbol findings above plus several React lint issues that were outside the requested audit scope.
- `npx tsc --noEmit` failed with a Next-generated typed-routes validator error in `.next/types/validator.ts` for `/proposal/[publicSlug]`.
- `npm run build` compiled the app bundle but then aborted with `spawn EPERM` while running the post-compile TypeScript/build phase in this environment, so I did not count that sandbox failure as a source-level defect by itself.
